Scenario-First Authoring Vision v1

Это целевая модель продукта: пользователь не начинает с raw blocks. Он начинает с проекта и root-сценария Main, кидает в него понятные модули, а потом при росте сложности выделяет child sequences. Под капотом система materialize-ит всё в реальные `signals + blocks + sequences`, но прошивка остаётся одной и той же.

Главная мысль

  • `Main` должен существовать всегда и быть безопасной стартовой точкой.
  • Простой проект может целиком жить в `Main` без sequence.
  • Когда логика стала фазовой, часть Main выделяется в child sequence, а не ломает уже сделанную работу.
  • Sequence обязан публиковать `enabled / active / waiting / done / fault / current_state`.
  • Модули должны быть меньше текущих “больших шаблонов” и ближе к composable bricks.
  • Blocks остаются честным advanced-слоем, но не главным путём входа.

Правило роста

Логика не должна заставлять пользователя “начать заново правильно”. Правильная система позволяет начать просто, а потом разрезать Main на более структурные child sequences без потери уже настроенных ресурсов и модулей.

Inspector / Wizard

Step 1. Bind
  • найти существующие channels/signals
  • если их нет, предложить создать
Step 2. Configure
  • режим модуля
  • timers / thresholds / policy
Step 3. Preview
  • какие signals будут созданы
  • какие blocks будут созданы
  • нужен ли owned sequence
Step 4. Publish
  • какие outputs пойдут наружу
  • что увидят Service / Display

Что реально исполняется

Сценарии и модули нужны человеку. Контроллер не должен исполнять их рекурсивно. После materialize у нас остаётся плоский runtime graph.

Source / Bound Signals
  • `compressor_running`
  • `manual_override`
  • `pressure_pv`
Generated Blocks
  • `timer.cycle_1`
  • `logic_gate.main_or_1`
  • `threshold.main_cmp_1`
Owned Sequences
  • `sequence.purge`
  • `sequence.burning`
  • plus alarms / outputs

Ключевое правило

Authoring graph может быть многоуровневым, но runtime graph должен оставаться плоским. Это и позволяет одной прошивке тянуть и простой проект, и сложный механизм.

К чему идём

  • Пользователь начинает с `Project -> Main`, а не с таблицы блоков.
  • Основной авторинг идёт через небольшие модули-кирпичики.
  • Когда нужна фазовая логика, пользователь добавляет child sequences.
  • Каждый module/sequence materialize-ит реальные internals.
  • Service и Display видят published outputs и выбранные generated values.
  • Одна и та же прошивка остаётся годной и для простого, и для сложного проекта.
