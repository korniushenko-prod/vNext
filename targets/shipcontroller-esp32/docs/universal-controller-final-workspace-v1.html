Что это
Не просто timer block. Это stateful authoring object, который материализуется в runtime-visible sequence/subflow с published status outputs.

Published outputs
`relay_cmd` `phase_state` `phase_remaining` `ready` `status_text`
Почему это лучше, чем голый click -> timer
Клик остаётся событием, а не должен висеть в true. Runtime хранит состояние между проходами сам. Debug может честно объяснить текущую фазу. Из этого легко вырастает общий sequence editor.

Roadmap vs ТЗ

Уже совпадает
  • `state machine` как верхний orchestration layer уже есть в документах и runtime.
  • `visual editor` уже правильно движется к `Project / Main / Inspector / Generated Internals`.
  • `subflows` уже концептуально поддержаны через child sequence и flow inside state.
  • `runtime` у вас уже stateful и умеет держать timers, blocks, sequences и published signals.
  • `JSON` уже оформлен как `project_model_v2` и как runtime-config путь.
Чего ещё не хватает до вашего ТЗ
  • Прямого user-facing узла `stateful command sequence` между event и output.
  • Явного разделения `event -> memory -> phase runtime -> actuator` в редакторе как стандартного паттерна.
  • Нормального explain/debug для причин `OFF because cooldown active` на уровне сценария.
  • Полноценного compiler/materialize path из редактора без ощущения bridge/демо-режима.
  • Библиотеки готовых сценарных паттернов поверх существующих primitive blocks.
Что лучше оставить как advanced
  • Raw `Blocks / Signals / Channels` не убирать, но держать под основным editor-first слоем.
  • Generated internals preview не тащить на первый экран по умолчанию.
  • Boiler не делать главным абстрактным языком продукта.
  • Свободный generic canvas не ставить раньше scenario-first flow editor.
  • Глубокие object-level настройки оставлять для service/advanced path.
