# Signal Source Node v1

## Назначение

`Signal Source` — это базовый узел-источник для editor-first модели.

Он отвечает не за "вход GPIO" как таковой, а за вопрос:

- откуда логика берёт сигнал
- какого типа этот сигнал
- что именно узел выдаёт дальше в flow

## Модель v1

Узел имеет 3 главные части:

1. `Тип источника`
2. `Откуда взять signal`
3. `Что выдаёт`

## Поддержанные типы в v1

### 1. `Дискретный`

Примеры:

- сухой контакт
- тумблер
- дискретный вход

Выходы:

- `state`

Runtime mapping:

- direct signal use
- timer/logic работают от уровня

### 2. `Кнопка / событие`

Примеры:

- кнопка пуск
- панельная кнопка
- momentary push-button

Поведение v1:

- `Нажатие / отпускание`
- `Toggle по нажатию`

Выходы:

- `pressed`
- `released`
- `toggle_state` для toggle-режима

Runtime mapping:

- `button` block
- опционально `latch` для toggle-режима

## Следующие типы, но не v1

- `Аналоговый`
- `Импульсный`
- `Виртуальный`
- `Системный`

Они уже являются частью целевой концепции, но не обязательны для завершения `Test 1`.

## Binding path

### Для `Дискретный` и `Кнопка / событие`

Путь v1:

- использовать существующий `signal`
- или создать `signal` из реального `channel`

Это означает:

- если signal уже существует, пользователь выбирает его из списка
- если signal ещё нет, пользователь выбирает existing channel или создаёт новый channel
- после этого signal привязывается к source-node

## UI contract

В инспекторе должны быть видны:

- `Тип источника`
- `Поведение`
- `Откуда взять signal`
- `Что выдаёт`

Валидация:

- обязательные поля подсвечиваются inline
- valid = зелёный
- invalid = красный
- optional поля нейтральны

## Для Test 1

`Test 1` использует этот узел как первый эталон editor-first authoring.

Минимально достаточно:

- `Дискретный`
- `Кнопка / событие`
- inline validation
- materialize в текущий runtime
