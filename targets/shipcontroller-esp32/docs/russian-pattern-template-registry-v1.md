# Russian Pattern / Template Registry v1

## Purpose

This document defines the first Russian-first pattern/template registry for the core automation groups.

The goal is to make the future UI speak in the language of tasks, not internal architecture terms.

This registry is intended for:

- module library
- module inspector
- guided setup
- future help panels
- future service and commissioning UX

It does not replace the underlying technical registries.

It maps them into user-facing language.

## Core Help Shape

Every user-facing pattern or template should expose at least:

- `Название`
- `Кратко`
- `Что делает`
- `Когда использовать`
- `Что нужно привязать`
- `Что выдаёт`
- `Типичные ошибки`

Optional later:

- `Что внутри`
- `Что можно раскрыть в advanced`
- `Какие alarms обычно рядом`

## Core Pattern Groups

## 1. Измеряемая величина

Technical family:

- `measured_value`

Examples:

- температура
- давление
- уровень
- расход
- проводимость / солёность
- состояние дискретного датчика

### Русское описание

- `Название`: `Измеряемая величина`
- `Кратко`: `Получает реальное значение из датчика или внешнего устройства и превращает его в нормализованный сигнал для автоматики.`
- `Что делает`:
  - читает значение
  - оценивает качество
  - может проверять рабочий диапазон
  - публикует нормализованный результат
- `Когда использовать`:
  - когда нужно взять температуру, давление, уровень, расход, TDS или состояние дискретного датчика
- `Что нужно привязать`:
  - источник сигнала или ресурс
  - единицы измерения
  - при необходимости рабочий диапазон
- `Что выдаёт`:
  - `value`
  - `ready`
  - `fault`
  - `quality`
  - `in_range`
- `Типичные ошибки`:
  - не тот источник
  - неправильные единицы
  - не настроен диапазон
  - stale/comm-loss не учтён

## 2. Условие / Порог / Окно

Technical family:

- `condition_decision`

Examples:

- выше порога
- ниже порога
- внутри окна
- вне окна
- равно / не равно

### Русское описание

- `Название`: `Условие / Порог / Окно`
- `Кратко`: `Проверяет значение и выдаёт логические состояния для автоматики.`
- `Что делает`:
  - сравнивает входное значение с порогом или диапазоном
  - может выдавать `above / below / in_window / out_of_window`
- `Когда использовать`:
  - если нужно принять решение по значению датчика
- `Что нужно привязать`:
  - входное значение
  - пороги или границы окна
  - режим сравнения
- `Что выдаёт`:
  - логические выходы условия
  - иногда summary статуса
- `Типичные ошибки`:
  - перепутаны верхний и нижний пороги
  - не учтён шум сигнала
  - нет hysteresis там, где он нужен

## 3. Гистерезис / Антидребезг значения

Technical family:

- `hysteresis_deadband`

### Русское описание

- `Название`: `Гистерезис / Deadband`
- `Кратко`: `Защищает условие от дрожания около порога.`
- `Что делает`:
  - задаёт раздельные условия включения и выключения
  - делает переключения устойчивыми
- `Когда использовать`:
  - если аналоговый сигнал шумит около порога
  - если нежелательны частые переключения
- `Что нужно привязать`:
  - входное значение
  - порог включения
  - порог выключения
  - режим
- `Что выдаёт`:
  - устойчивое логическое состояние
- `Типичные ошибки`:
  - слишком маленький разрыв
  - перепутаны уровни on/off

## 4. On/Off Контур

Technical family:

- `onoff_control`

Examples:

- пуск насоса по уровню
- останов по давлению
- циклическое включение продувки

### Русское описание

- `Название`: `On/Off Контур`
- `Кратко`: `Включает и выключает исполнительный механизм по условиям, гистерезису и таймерам.`
- `Что делает`:
  - формирует запрос `включить / выключить`
  - использует пороги, hysteresis, min on/off, delay
- `Когда использовать`:
  - когда нужен не PID, а простой устойчивый on/off контроль
- `Что нужно привязать`:
  - источник условия
  - пороги/окно
  - выходной механизм или command target
- `Что выдаёт`:
  - `request`
  - `active`
  - `waiting_reason`
  - `fault` при необходимости
- `Типичные ошибки`:
  - нет min on/off
  - не учтено дрожание уровня
  - перепутан start/stop policy

## 5. PID Контур

Technical family:

- `pid_control`

### Русское описание

- `Название`: `PID Контур`
- `Кратко`: `Поддерживает заданное значение по непрерывной регулирующей логике.`
- `Что делает`:
  - сравнивает PV и SP
  - формирует MV
  - управляет модулем/выходом
- `Когда использовать`:
  - температура
  - давление
  - модульная заслонка
  - дозирование
- `Что нужно привязать`:
  - `PV`
  - `SP`
  - target управления
  - пределы выхода
- `Что выдаёт`:
  - `mv`
  - `control_active`
  - `fault`
  - `ready`
- `Типичные ошибки`:
  - нет границ выхода
  - неверные единицы
  - плохая настройка коэффициентов

## 6. Исполнительный механизм

Technical family:

- `actuator`

Examples:

- реле насоса
- вентилятор
- соленоид
- реверсивный клапан
- позиционер
- PWM-нагрузка

### Русское описание

- `Название`: `Исполнительный механизм`
- `Кратко`: `Преобразует логическую команду в реальное управление нагрузкой или приводом.`
- `Что делает`:
  - управляет выходом
  - отслеживает feedback
  - публикует состояние и fault
- `Когда использовать`:
  - для насосов, клапанов, вентиляторов, соленоидов, заслонок, нагревателей
- `Что нужно привязать`:
  - command target
  - feedback при наличии
  - capability режима
- `Что выдаёт`:
  - `running`
  - `busy`
  - `ready`
  - `fault`
  - `position` или `state`
- `Типичные ошибки`:
  - capability не совпадает с реальным выходом
  - feedback не привязан
  - timeout не настроен

## 7. Счётчик / Totalizer

Technical family:

- `totalizer_accumulator`

Examples:

- импульсный расходомер
- счётчик циклов
- runtime counter

### Русское описание

- `Название`: `Счётчик / Totalizer`
- `Кратко`: `Считает импульсы, накапливает totals и формирует расчётные накопленные значения.`
- `Что делает`:
  - считает события или импульсы
  - ведёт totals
  - может хранить session/resettable/lifetime значения
- `Когда использовать`:
  - flowmeter
  - счётчик циклов
  - наработка
- `Что нужно привязать`:
  - вход импульса или события
  - масштаб
  - политику сохранения
- `Что выдаёт`:
  - `count`
  - `session_total`
  - `resettable_total`
  - `lifetime_total`
- `Типичные ошибки`:
  - считают по уровню, а не по фронту
  - слишком частая запись в память
  - нет scale/units

## 8. Извлечение импульсов / Скорости / Настройка

Technical family:

- `signal_extraction_tuning`

Reference:

- `docs/pulse-rate-extraction-pattern-v1.md`

### Русское описание

- `Название`: `Извлечение импульсов / Скорости / Настройка`
- `Кратко`: `Преобразует сырой или аналоговый сигнал в устойчивые события, частоту, скорость или инженерную величину с удобной наладкой.`
- `Что делает`:
  - кондиционирует сигнал
  - применяет пороги и hysteresis
  - извлекает фронты/импульсы
  - может считать скорость, частоту или инженерный расход
  - показывает live debug
  - помогает подобрать уставки
- `Когда использовать`:
  - flowmeter
  - RPM вентилятора
  - RPM двигателя
  - аналоговый датчик, который нужно превратить в цифровое условие
  - извлечение импульсов
  - настройка порогового поведения
- `Что нужно привязать`:
  - источник или два источника
  - режим extraction
  - пороги
  - hysteresis
  - output profile
  - edge policy
- `Что выдаёт`:
  - conditioned signal
  - digital state
  - edges/events
  - count/rate/engineering value в зависимости от профиля
  - debug summary
- `Типичные ошибки`:
  - слишком узкий hysteresis
  - пороги выбраны без live debug
  - считают шум как импульсы
  - показывают flowmeter-поля там, где задача только про RPM

## 9. Последовательность / Фазы

Technical family:

- `sequence`

### Русское описание

- `Название`: `Последовательность / Sequence`
- `Кратко`: `Ведёт механизм по фазам и переходам с таймерами, причинами ожидания и fault-логикой.`
- `Что делает`:
  - переключает состояния
  - проверяет условия переходов
  - ведёт timeout и причины
- `Когда использовать`:
  - запуск/останов
  - purge
  - ignition
  - cooldown
  - wash cycle
- `Что нужно привязать`:
  - входные сигналы условий
  - actions
  - таймеры
  - transition policy
- `Что выдаёт`:
  - `running`
  - `done`
  - `fault`
  - `current_state`
  - `waiting_reason`
- `Типичные ошибки`:
  - sequence используют там, где хватило бы on/off контура
  - не определены fault transitions
  - не описаны reset/restart semantics

## 10. Alarm / Trip Политика

Technical family:

- `alarm_policy`

### Русское описание

- `Название`: `Alarm / Trip Политика`
- `Кратко`: `Собирает предупреждения, trip-события и lockout-поведение вокруг механизма или функции.`
- `Что делает`:
  - различает warning и trip
  - может latch/ack
  - публикует состояние аварий
- `Когда использовать`:
  - почти в любом реальном механизме
- `Что нужно привязать`:
  - источники faults/trips/warnings
  - severity/policy
- `Что выдаёт`:
  - `alarm`
  - `trip`
  - `lockout`
  - service/event summary
- `Типичные ошибки`:
  - warning ошибочно делают trip
  - все alarms смешаны в один слой без политики

## Immediate UI Rule

The first UI pass should not try to expose every possible template.

It should start with these core user-facing groups:

1. `Измеряемая величина`
2. `Условие / Порог / Окно`
3. `On/Off Контур`
4. `PID Контур`
5. `Исполнительный механизм`
6. `Счётчик / Totalizer`
7. `Извлечение импульсов / Скорости / Настройка`
8. `Последовательность / Sequence`
9. `Alarm / Trip Политика`
