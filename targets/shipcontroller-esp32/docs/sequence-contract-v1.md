# Sequence Contract v1

## Purpose

This document fixes the behavioral contract for `sequence` as a first-class runtime object.

It exists to keep sequence:

- non-blocking
- explainable
- module-friendly
- safe to visualize later

It also prevents sequence from drifting into:

- a hidden pile of autogenerated ordinary blocks
- a vendor-specific macro system
- an opaque UI-only concept

## Core Rule

A sequence is a non-blocking state machine updated from the shared main loop.

A sequence must:

- read inputs
- evaluate transitions
- evaluate timers
- publish state/status outputs
- optionally issue actions

all within one short loop step.

It must never:

- block waiting for feedback
- sleep/delay inside a state
- monopolize the CPU until a step completes

## Sequence Runtime Shape

Every sequence instance conceptually owns:

1. `inputs`
2. `states`
3. `transitions`
4. `timers`
5. `actions`
6. `runtime state`
7. `published outputs`
8. `service reasons`

### Inputs

Inputs are normalized signal/module inputs such as:

- start
- stop
- reset
- permissive
- trip
- mode
- feedback

### States

Each state should have stable meaning such as:

- `idle`
- `precheck`
- `warmup`
- `wait_feedback`
- `run`
- `cooldown`
- `done`
- `fault`

### Transitions

Transitions should be explicit objects with:

- source state
- target state
- condition
- delay
- optional invert/edge semantics
- optional reason text

### Timers

Timers are per-state or per-transition elapsed-time checks.

They must remain non-blocking.

### Actions

Actions may target:

- `channel:<id>`
- `signal:<id>`
- `command:<id>`

Actions must stay explicit and explainable from editor and service views.

### Runtime State

Runtime state should minimally include:

- current state id
- previous state id
- state-enter timestamp
- running/done/fault flags
- pending transition data

### Published Outputs

Sequence must publish dependent service/status signals automatically.

Recommended outputs include:

- `running`
- `ready`
- `waiting`
- `done`
- `fault`
- `current_state`
- `current_state_index`
- `time_in_state_ms`
- `state.<stateId>`

### Service Reasons

A usable sequence must explain itself.

Recommended service-facing fields:

- `waiting_reason`
- `fault_reason`
- `detail`
- `pending_transition_signal`
- `pending_transition_delay_ms`
- `pending_transition_elapsed_ms`

## Mandatory Behavioral Rules

### Rule A: Sequence Owns Orchestration, Not Hardware Discovery

Sequence may target channels or command signals, but it must not auto-create physical channels.

### Rule B: Sequence Publishes Its Own Dependent Signals

The user should not need to manually recreate basic sequence status through ordinary blocks.

### Rule C: Sequence Remains Source Of Truth

If a future visual editor exists, it must edit the same sequence model.

No second hidden runtime is allowed.

### Rule D: Transitions Are Evaluated Incrementally

Every loop step may:

- stay in state
- progress delay timing
- change state
- enter fault/done

but it must not run an internal blocking mini-loop.

### Rule E: Reasons Are First-Class

Waiting and fault explanations are not optional polish.

They are part of the contract needed for:

- service UX
- operation mimic
- future visual authoring

## Sequence And Modules

Sequence should operate through module interfaces whenever possible.

That means sequence should prefer:

- normalized module outputs as conditions
- normalized command targets as actions

instead of binding itself to vendor-specific transport details.

## Sequence And Parallelism

The platform should not model parallel mechanism behavior as spawned OS-like processes.

Instead:

- multiple sequences and modules coexist
- each updates once per control-loop pass
- coordination happens through signals, module outputs and transition rules

This is the approved non-blocking parallelism model for the platform.
